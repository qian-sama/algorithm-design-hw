# 0/1 Knapsack（DP）实验结果分析

## 实验环境与数据集

- **项目**：`knapsack_dp`（C++）
- **算法**：0/1 背包动态规划（1D 滚动数组）+ 路径回溯
- **数据集**：`Datasets-Course/4.1 Florida State University`（p01–p08）
- **运行方式**（示例）：

```bash
./build/knapsack_dp.exe "Datasets-Course/4.1 Florida State University"
```

## 实现要点（与代码一致）

- **状态定义**：`dp[c]` 表示容量为 \(c\) 时可达到的最大收益。
- **转移**：对每个物品 \(i\)，从 `c = C..w[i]` 逆序更新：
  \[
  dp[c] = \max(dp[c],\ dp[c-w_i] + v_i)
  \]
- **空间优化**：使用一维数组，因此空间为 \(O(C)\)。
- **解回溯**：使用 `path[c]`（最后一次选择的物品下标）和 `from_cap[c]`（来自的容量）从 `c=C` 反向恢复一个最优解。
- **容量上限保护**：当 `capacity > max_capacity(默认 1,000,000)` 时直接返回 `SKIP`（避免 DP 资源爆炸）。

## 结果分析

- **正确性**：p01–p07 的 `DP profit` 与 `optimal profit` 全部一致，说明实现的转移与回溯逻辑正确，且数据读取无误。
- **运行时间**：在 p01–p07 上耗时均为亚毫秒级到 0.1ms 级，符合 \(O(nC)\) 的规模预期（这些实例的容量 \(C\) 较小）。
- **p08 被跳过的原因**：p08 的容量 \(C=6,404,180\)；若直接做 DP，需要长度约 640 万的数组，且时间大约为 \(nC \approx 24 \times 6.4\text{M} \approx 1.5 \times 10^8\) 次更新，资源开销显著。项目用 `max_capacity` 做保护，避免在普通机器上长时间运行或内存压力过大，因此返回 `SKIP` 是预期行为。

## 优点与缺点

### 优点

- **保证最优解**：经典 0/1 背包 DP 对可承受容量范围内的实例给出全局最优。
- **空间优化**：1D 滚动数组将空间从 \(O(nC)\) 降到 \(O(C)\)。
- **可输出一个最优选择方案**：通过 `path/from_cap` 回溯得到选取向量（不仅仅是最大收益）。
- **实现稳健**：对容量过大（或非法输入）返回可解释的失败信息，避免“跑崩”。

### 缺点

- **对容量敏感**：时间/空间都随 \(C\) 线性增长，属于“伪多项式”算法；当 \(C\) 很大时不适用（如 p08）。
- **依赖整数容量/权重**：这种 DP 需要容量维度是可枚举的整数；若容量尺度大或非整数，直接 DP 不现实。
- **回溯方案非唯一**：当存在多个最优解时，`path` 记录的是更新路径之一，只会恢复到其中一个最优解。

## 改进建议（面向大容量实例）

- **分支限界 / 回溯剪枝**：对大容量但物品数不算特别大的情况，可能比 DP 更合适（取决于数据结构与剪枝效果）。
- **基于价值维度的 DP（最小重量 DP）**：当总价值规模更可控时，可将维度从容量换成价值。
- **近似算法（FPTAS）**：需要在大规模实例上给出可控误差的近似最优解时使用。
